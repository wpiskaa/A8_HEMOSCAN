BAB 1: TEORI DASAR RELATIONAL DATABASE (RDBMS)

1.1. DDL (Data Definition Language) & DML (Data Manipulation Language)

Dalam SQL, perintah dibagi menjadi kategori berdasarkan tujuannya:

  • DDL (Mendefinisikan Struktur): CREATE (membuat tabel/database), ALTER (mengubah struktur tabel), DROP (menghapus tabel/database). Di HemoScan, DDL digunakan di file query_hemoscan.sql saat mendefinisikan tabel.
  • DML (Memanipulasi Data): INSERT, UPDATE, DELETE, dan SELECT. Di HemoScan, DML dieksekusi dari dalam kode C# (ADO.NET).

1.2. Normalisasi Basis Data (Mencapai 3NF)

Desain database HemoScan telah memenuhi kaidah Normalisasi tingkat 3 (3NF):

  1. 1NF: Tidak ada grup berulang atau kolom multivalue. Setiap sel berisi satu nilai (misal: Rhesus dipisah dari Golongan Darah).
  2. 2NF: Sudah memiliki Primary Key yang absolut (ID_Kantong), dan tidak ada ketergantungan parsial.
  3. 3NF: Tidak ada atribut yang bergantung pada atribut non-kunci lainnya (Ketergantungan Transitif). Nama Unit Medis dan Alamat tidak ditaruh di Tabel_Kantong_Darah, melainkan dipisah ke Tabel_Unit_Medis lalu dihubungkan dengan Foreign Key.

1.3. Integritas Data (Data Integrity)

PABD mewajibkan data tetap konsisten dan valid. HemoScan menerapkan 3 jenis integritas:

  • Entity Integrity: Diterapkan melalui PRIMARY KEY. Menjamin tidak ada dua baris yang memiliki ID yang sama persis dan ID tidak boleh NULL.
  • Referential Integrity: Diterapkan melalui FOREIGN KEY. Contoh: ID_Unit di Tabel_Kantong_Darah harus merujuk ke ID_Unit yang benar-benar ada di Tabel_Unit_Medis.
  • Domain Integrity: Diterapkan melalui CHECK Constraint. Contoh: CHECK (Gol_Darah IN ('A', 'B', 'AB', 'O')). Ini mencegah human error.

1.4. Operasi JOIN & Agregasi

Untuk menampilkan informasi yang ramah manusia (menampilkan "RS Bethesda", bukan ID "2"), kita menggunakan INNER JOIN, yang hanya menampilkan baris yang memiliki pasangan di kedua tabel.

HemoScan juga menggunakan fungsi agregasi SQL untuk dashboard: COUNT(*) untuk menghitung jumlah total stok, dan GROUP BY untuk mengelompokkan stok berdasarkan golongan darah.

BAB 2: TEORI ADO.NET (KONEKSI C# KE SQL SERVER)

2.1. Connected vs Disconnected Architecture

  • Connected (SqlDataReader): Sangat cepat, efisien memori, bersifat Forward-Only. Cocok untuk mengecek login pengguna.
  • Disconnected (SqlDataAdapter + DataTable): Data diambil ke memori (RAM), lalu koneksi diputus. Cocok untuk ditampilkan di DataGridView.

2.2. Eksekusi Perintah (The 3 Execution Methods)

  1. ExecuteNonQuery(): Untuk DML modifikasi data (INSERT, UPDATE, DELETE). Mengembalikan jumlah baris yang terpengaruh.
  2. ExecuteScalar(): Untuk mengambil persis SATU Niai tunggal (1 Baris, 1 Kolom), seperti SELECT COUNT(*).
  3. ExecuteReader(): Untuk query SELECT yang menghasilkan banyak baris data.

2.3. Keamanan ADO.NET (Parameterized Query)

Untuk mencegah SQL Injection, kita WAJIB menggunakan cmd.Parameters.AddWithValue. Ini membuat database memperlakukan input pengguna secara murni sebagai "teks data", bukan sebagai perintah SQL jahat.

BAB 3: BEDAH ALGORITMA HEMOSCAN (CHANGELOG TERBARU)

3.1. Algoritma Pemrosesan Request (Integrasi Stok dan Transaksi)

Ini adalah "Jantung" dari integritas bisnis aplikasi. Saat PMI memproses Request, aplikasi TIDAK HANYA mengubah status Request. Alur logikanya adalah:

1. Pemisahan String: Memecah input "A+" menjadi Golongan "A" dan Rhesus "+".
2. Validasi Ketersediaan: Mengeksekusi SELECT TOP 1 ID_Kantong FROM Tabel_Kantong_Darah WHERE Gol_Darah = 'A' AND Rhesus = '+' AND Status = 'Tersedia'.
3. Pengecekan: Jika stok kosong, batalkan proses.
4. Update Stok Fisik: Ubah Status = 'Dikirim' HANYA untuk 1 ID_Kantong tersebut.
5. Update Request: Ubah status request menjadi 'Dikirim'.

3.2. Pencegahan Anomali Modifikasi (UI/UX Lock)

Di form Admin PMI, jika kita memilih kantong darah yang statusnya sudah 'Dikirim', tombol Update dan Hapus akan otomatis mati (Disabled). Barang yang sudah dikirim secara fisik tidak boleh diedit atau dihapus secara sepihak untuk menjaga jejak audit medis.
